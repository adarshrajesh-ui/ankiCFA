## 0. Framing & the honesty rule (read first)

This is a **CFA Level II exam-prep app built on top of Anki** — not "Anki with a
CFA tab." It must answer three *separate* questions with ranges: **Memory**,
**Performance**, **Readiness**.

**Honesty rule (spec, non-negotiable):** never print or display a
readiness/eval number without the evidence behind it. Any metric computed on
synthetic/simulated data MUST be labelled `SIMULATED` in its output and docs.
Keep the give-up rule (abstain when data is insufficient). The app must still
produce a score with **AI switched off**. Fabricating or dressing up a guess as
a measurement is an automatic fail — do not do it.

**Deadline focus = FRIDAY:** "AI is added and checked; phone syncs with desktop
and shows readiness." P0 items below are the Friday-critical gaps. P1 items are
high grading-weight gaps. P2 is Sunday/packaging (handled outside these loops).

## 1. Global rules for every GNHF worker

1. **Do NOT run `no-mistakes`.** Instead, add 1–3 fast tests per item and run the
   targeted `just cfa-*` recipe or `pytest`/`cargo test`. Keep iterations tight.
2. **Do NOT commit to `main`.** Work only on your own gnhf branch/worktree.
   Commit after each finished item with a clear message.
3. **Stay strictly in your SCOPE** (see your loop section). The only shared file
   is `justfile`: append new recipes at the **very end** under a banner comment
   `# --- <LOOP> speedrun recipes ---` so merges stay trivial.
4. **Keep every currently-green test green.** If you touch scores/topics, update
   the affected tests and re-run `just cfa-scores-test cfa-parity-test cfa-f4-test`.
5. **Prefer targeted tests over full builds.** Avoid `just check` / full `just
   build` (slow, and other loops may be building). Use `PYTHONPATH=out/pylib:pylib
   pytest <file>` and the `cfa-*` recipes. A web rebuild (`just rebuild-web`) or
   `cargo test -p anki <mod>` is fine when needed.
6. **Every item = committed code + a passing test + an evidence file** under
   `proof/friday/gnhf-speedrun/<LOOP>/`. Log status in that loop's `NOTES.md`.
7. **If blocked:** write `BLOCKED: <root cause>` in your NOTES, cite the closest
   proof you *can* produce, then move to the next item. Never fake success.
8. Before coding, `git log --oneline -15` and skim the touched files — preserve
   existing user changes; no unrelated refactors.
